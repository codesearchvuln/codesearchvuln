# VulHunter Deployment Guide

This package is used to deploy VulHunter with Docker Compose. It includes the startup configuration, environment templates, web UI static assets, and helper scripts required to launch the web interface, backend service, database, Redis, and the default runner components.

## 1. Quick Start

Copy the backend environment template before the first run:

```bash
cp docker/env/backend/env.example docker/env/backend/.env
```

Then set at least:

- `LLM_API_KEY`
- `LLM_PROVIDER`
- `LLM_MODEL`

If your deployment needs database, Redis, auth, or other runtime settings, update `docker/env/backend/.env` accordingly.

Once configured, start VulHunter with:

```bash
docker compose up -d
```

## 2. Supported Deployment Scope and Disclaimers

- This release package is intended for host-based deployment only. Supported host environments are:
  `Ubuntu 22.04 LTS`, `Ubuntu 24.04 LTS`, `Windows 10`, `Windows 11`, `Windows 10 WSL2 + Ubuntu 22.04 LTS`, and `Windows 11 WSL2 + Ubuntu 22.04 LTS`.
- On `Windows 10/11`, use `Docker Desktop` with Linux containers enabled. The release runtime depends on Docker Engine / Docker Compose, and the backend container must be able to access the Docker socket (default `/var/run/docker.sock`) to run runner preflight checks and actual scan jobs. If your Docker socket path or group ID differs from the default, set `DOCKER_SOCKET_PATH` and `DOCKER_SOCKET_GID` yourself. Problems caused by Docker Desktop Linux container mode, WSL integration, Docker socket mounting, or permission / group ID issues are outside the support scope of this release.
- The current release tree is generated by cloud GitHub Workflows and is an image-only runtime package. It ships runtime `docker-compose.yml`, env templates, static assets, and helper scripts only. It does not include `backend` / `frontend` source code, does not support rebuilding `backend` / `frontend` inside the release package, and does not provide adaptations for non-Docker deployment, Kubernetes, running from source, or other derived deployment models. Under this release contract, runner preflight only validates and pulls the declared runtime images; it does not fall back to local builds.
- The release tree now ships both Bash and PowerShell offline helper scripts. Offline mode additionally requires `docker` and `zstd`; the Bash variant also requires `python3`. On `Windows 10/11`, you can run the bundled native PowerShell scripts directly, or keep using the Bash scripts from WSL.
- The recommended runtime baseline is `8 CPU cores` and `16 GB RAM`. Below that level, image pulls, runner preflight checks, scan execution, and LLM interactions may become significantly slower or fail, so running the system is not recommended.
- Supported browsers are `Safari`, `Chrome`, and `Edge`. The frontend relies on browser capabilities such as `EventSource/SSE`, `iframe`, `localStorage`, and `clipboard`. For stable behavior, disable all browser extensions / plugins. Issues caused by extensions, content blockers, enterprise security plugins, or unsupported browsers will not receive additional compatibility work.
- Network access is optional, but online deployment requires network connectivity to pull runtime images, and cloud LLM integrations require the target API endpoints to be reachable. Offline deployment only replaces image pulls; it does not replace cloud model APIs. If you want offline startup, use the offline image bundles that match the current machine architecture (`amd64` or `arm64`).
- The stack exposes `3000`, `8000`, `5432`, and `6379` by default, plus `8081` under the `tools` profile. Port conflicts, firewall policies, local security software, or enterprise network controls must be handled by the deployer.
- Runtime data is persisted in Docker volumes such as `postgres_data`, `backend_uploads`, `backend_runtime_data`, `scan_workspace`, and `redis_data`. Removing containers or running `docker compose down -v` deletes related data, so backup, migration, and retention policies are the deployer's responsibility.
- Host environments, virtualization setups, container runtimes, browser combinations, and proxy / mirror strategies outside the list above have not been validated as part of the current release contract. If you run the package in undeclared environments, you must diagnose and resolve resulting issues yourself.

## 3. Configuration Overview

`docker/env/backend/.env` is the main runtime configuration file for VulHunter. Typical settings include:

- LLM provider and model
- API keys and endpoints
- database connection settings
- Redis connection settings
- scanner feature flags and image overrides

For a minimal first deployment, `LLM_API_KEY`, `LLM_PROVIDER`, and `LLM_MODEL` are usually enough to get started.

## 4. Online Deployment (Default)

```bash
docker compose up -d
```

By default, VulHunter pulls the required runtime images and starts the full stack. The main web UI static assets and the default nginx configuration are already included in this package; `STATIC_FRONTEND_IMAGE` provides the nginx base image that serves them. The deployment starts `db` and `redis`. The `nexus-web` / `nexus-itemDetail` pages are now served by mounting the local static bundles into the main frontend container instead of starting dedicated containers. Code execution now uses a single `SANDBOX_RUNNER_IMAGE`; the release contract no longer exposes a separate legacy sandbox image override.

## 5. Offline Deployment (Optional)

Download both matching offline bundles for your machine:

- `vulhunter-services-images-<arch>.tar.zst`
- `vulhunter-scanner-images-<arch>.tar.zst`

Place them in the release root or the `images/` directory, then run:

```bash
cp docker/env/backend/offline-images.env.example docker/env/backend/offline-images.env
./scripts/load-images.sh
./scripts/use-offline-env.sh docker compose up -d
```

On native `Windows 10/11 PowerShell`, you can run:

```powershell
Copy-Item docker/env/backend/offline-images.env.example docker/env/backend/offline-images.env
powershell -ExecutionPolicy Bypass -File .\scripts\load-images.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\use-offline-env.ps1 docker compose up -d
```

Offline mode imports both `services` and `scanner` bundles first and then switches runtime services to local `vulhunter-local/*` tags, so the stack can start without pulling images from the network. The web UI and `nexus-*` static assets are still served from the files shipped in this package and are not included in the offline image bundles.

## 6. Run and Maintain

Follow logs:

```bash
docker compose logs -f
```

Stop the stack:

```bash
docker compose down
```

Stop and remove volumes:

```bash
docker compose down -v
```

After changing `.env` or `offline-images.env`, rerun `docker compose up -d` to apply the update.

## 7. Common Image Overrides

If you need to point the stack to custom images or an existing runtime environment, you can override these values in `docker/env/backend/.env`:

- `BACKEND_IMAGE`
- `POSTGRES_IMAGE`
- `REDIS_IMAGE`
- `ADMINER_IMAGE`
- `SCAN_WORKSPACE_INIT_IMAGE`
- `STATIC_FRONTEND_IMAGE`
- `SCANNER_*_IMAGE`
- `FLOW_PARSER_RUNNER_IMAGE`
- `SANDBOX_RUNNER_IMAGE`

`STATIC_FRONTEND_IMAGE` controls the nginx base image used to serve the packaged web UI static assets.

## 8. Service Endpoints

- Frontend: `http://localhost:3000`
- Backend API: `http://localhost:8000`
- OpenAPI docs: `http://localhost:8000/docs`

See [`scripts/README-COMPOSE.md`](scripts/README-COMPOSE.md) for Docker Compose operations and maintenance details.
- `nexus-web`: `http://localhost:${VULHUNTER_FRONTEND_PORT:-3000}/nexus/`
- `nexus-itemDetail`: `http://localhost:${VULHUNTER_FRONTEND_PORT:-3000}/nexus-item-detail/`
